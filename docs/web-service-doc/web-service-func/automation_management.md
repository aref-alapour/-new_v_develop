# automation_management

## تابع در web-service.php

```php
function automation_management($product_id, $booking_time = '')
```

## کارایی

- **نقش:** برای **یک product_id خاص (۲۸۳۲۵)** یک لاجیک سفارشی اجرا می‌کند: با cURL و پروکسی به آدرس ثابت t4f.ir درخواست می‌زند، از پاسخ HTML با regex سانس‌ها (data-status و data-start-at) را استخراج می‌کند و با `logintotag` خروجی را لاگ می‌کند. برای هر product_id دیگر در حال حاضر کاری انجام نمی‌دهد.
- **ورودی:**  
  - `$product_id`: عدد؛ فقط اگر برابر ۲۸۳۲۵ باشد بدنه اجرا می‌شود.  
  - `$booking_time`: در امضا هست ولی در بدنه استفاده نمی‌شود.
- **خروجی:** خروجی مستقیم ندارد؛ فقط با `logintotag($result)` یا `logintotag(['error' => ...])` لاگ می‌کند و در صورت نیاز هدر JSON و CORS ست می‌کند.
- **کجا صدا زده می‌شود:** در همین فایل در یک خط کامنت‌شده (~۱۲۹۶): `// automation_management($product_id);` — یعنی در وضع فعلی **اصلاً فراخوانی نمی‌شود** و تابع عملاً بیکار است.

## جایگزینی

- **حذف کامل:** چون جایی صدا زده نمی‌شود، می‌توانی کل تابع را حذف کنی. اگر فقط برای دیباگ/ادغام با سرویس t4f بود، بهتر است در یک اسکریپت جدا یا یک endpoint موقت ادمین قرار بگیرد، نه داخل web-service عمومی.
- **فعال کردن دوباره:** اگر خواستی دوباره استفاده شود، باید خط فراخوانی را از کامنت درآوری و مشخص کنی در چه نوع درخواستی (مثلاً یک type خاص در POST) صدا زده شود؛ و ترجیحاً به‌جای لاگ خام، خروجی را به فرمت یکسان API (مثلاً JSON) برگردانی و خطاها را هندل کنی.
- **جایگزین با سرویس جدا:** منطق «دریافت سانس از t4f» را می‌توانی در یک فایل/کلاس جدا (مثلاً `integrations/t4f_sync.php`) قرار بدهی و از web-service فقط در صورت نیاز آن را صدا بزنی؛ تا وابستگی و پروکسی و کوکی در یک جا جمع باشد.

## بهینه‌سازی

1. **پروکسی و کوکی:** آدرس، پروکسی، پورت و هدرهای کوکی الان هاردکد هستند؛ باید از config یا env خوانده شوند و کوکی‌ها به‌صورت داینامیک (مثلاً از session یا توکن) تنظیم شوند تا منقضی نشوند.
2. **خطای cURL:** بعد از `curl_exec` باید `curl_errno` و `curl_getinfo` را چک کنی؛ الان بعد از `curl_close` به `curl_getinfo` اشاره شده که اشتباه است و باید قبل از close باشد. در صورت خطا خروجی استاندارد (مثلاً JSON با پیام خطا) برگردان.
3. **Regex و ساختار پاسخ:** وابستگی به ساختار HTML و attributeهای دامنهٔ سوم را در docblock بنویس؛ در صورت تغییر سایت t4f، فقط یک نقطه (این تابع یا ماژول) نیاز به تغییر دارد.
4. **امنیت:** اگر این endpoint در web-service باز بماند، فقط برای product_id مشخص و ترجیحاً با توکن یا IP محدود فراخوانی شود تا سوءاستفاده نشود.
5. **خروجی:** به‌جای فقط `logintotag`، خروجی را به‌صورت آرایهٔ ساختاریافته برگردان (و در صورت نیاز با echo json_encode به caller بده) تا فرانت یا کرون بتواند از نتیجه استفاده کند.
