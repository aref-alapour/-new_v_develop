# helper/get_order_ids.php

**مسیر:** `wp-content/themes/escapezoom-v2/app/functions/helper/get_order_ids.php`  
**بارگذاری:** از طریق `app/functions/init.php` (include_once).

---

## خلاصه

دو تابع برای کار با سفارشات و محصولات ووکامرس: یکی گرفتن شناسه سفارشاتی که محصول(های) مشخص در آن‌ها با وضعیت(های) و بازه تاریخ و صفحه‌بندی، و دیگری گرفتن شناسه مالک/ثبت‌کننده محصول از متا.

---

## توابع

### 1) `get_orders_ids_by_product_id( $products_id, $order_status, $date_range = false, $limit = 10000000, $page = 1, $total = false )`

| پارامتر | نوع | توضیح |
|---------|-----|--------|
| `$products_id` | array | آرایه شناسه محصولات |
| `$order_status` | array | وضعیت‌های سفارش (مثلاً wc-completed, wc-processing) |
| `$date_range` | false یا array | [timestamp_start, timestamp_end] برای فیلتر تاریخ |
| `$limit` | int | تعداد در هر صفحه |
| `$page` | int | شماره صفحه |
| `$total` | bool | اگر true باشد به جای لیست order_id، تعداد کل برمی‌گردد |

**خروجی:**  
- اگر `$total === true`: نتیجه از یک زیرکوئری با COUNT و GROUP BY (تعداد رکوردها به ازای هر تاریخ)؛ در عمل خروجی به صورت آرایه از یک فیلد است.  
- اگر `$total === false`: آرایه شناسه سفارش‌ها (order_id) مرتب بر اساس تاریخ نزولی، با LIMIT و OFFSET.

**خطوط تقریبی:** ۲–۶۲.

**نکته امنیتی:**  
مقادیر `$products_id` و `$order_status` مستقیم داخل SQL با implode قرار می‌گیرند؛ باید قبل از فراخوانی با `absint` یا آرایه اعداد معتبر سانیتایز شوند تا از SQL injection جلوگیری شود. استفاده از `$wpdb->prepare` برای بخش داینامیک توصیه می‌شود.

---

### 2) `get_owner_id_by_product_id( $product_id )`

**خطوط تقریبی:** ۶۳–۶۵.

برگرداندن مقدار متای `user_ebtal` برای پست (محصول) با شناسه `$product_id`. معمولاً به عنوان «مالک/ثبت‌کننده محصول» استفاده می‌شود.

---

## استفاده در سایت

- گزارش فروش، لیست سفارشات مرتبط با یک یا چند محصول، فیلتر بر اساس تاریخ و وضعیت و صفحه‌بندی.
- در پنل یا وب‌سرویس برای نمایش سفارشات یک محصول/فروشنده.

---

## تداخل با helper/api.php

در `api.php` تابعی با همین نام `get_orders_ids_by_product_id` وجود دارد با منطق بسیار مشابه؛ تفاوت اصلی در بخش `$total` (ساختار کوئری و خروجی). بهتر است فقط یک نسخه در پروژه بماند (ترجیحاً این فایل) و همه جا از آن استفاده شود و نسخه api حذف یا به این تابع ارجاع داده شود.

---

## نحوه تغییر

- برای تغییر جداول/ستون‌ها: از پیشوند `$wpdb->prefix` استفاده شده؛ در صورت تغییر ساختار ووکامرس، نام جداول را تطبیق دهید.
- برای سانیتایز: قبل از فراخوانی، `$products_id` را به آرایه اعداد صحیح و `$order_status` را به مقادیر مجاز (مثلاً از لیست وضعیت‌های WC) محدود کنید و در صورت امکان کوئری را با prepare بنویسید.

---

## بهینه‌سازی پیشنهادی

1. کوئری را با `$wpdb->prepare` برای بخش‌های داینامیک (لیست product_id و order_status) بازنویسی کنید.
2. در حالت `$total` خروجی و منظور از «تعداد کل» را روشن کنید (یک عدد کل در کل بازه یا جمع تعداد به ازای هر روز) و در صورت نیاز خروجی را یک عدد واحد کنید.
3. یکسان‌سازی با نسخه api و حذف تکراری.
