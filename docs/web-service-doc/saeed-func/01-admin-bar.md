## disable_admin_bar_for_non_admins

- **تابع در `saeed-codes.php`:** `disable_admin_bar_for_non_admins()`
- **جایگزین چه تابعی شده؟**  
  در گذشته نسخه‌ای کامنت‌شده با نام `admin_bar()` بود که روی `init` هوک می‌شد و همیشه برای کاربر لاگین‌شده admin bar را می‌بست. تابع فعلی جایگزین آن شده و **رفتار را محدود کرده به نقش‌های مشخص** (`customer`, `compiler`, `sans_manager`).

### کارایی دقیق

- روی هوک `after_setup_theme` ثبت شده است.
- اگر کاربر نقش‌های بالا را داشته باشد، فیلتر `show_admin_bar` را روی `__return_false` قرار می‌دهد.
- نتیجه: برای مشتری‌ها و مدیران سانس/ساخت اتاق، نوار ادمین نمایش داده نمی‌شود ولی برای ادمین و نقش‌های دیگر باقی می‌ماند.

### پیشنهادهای بهینه‌سازی

1. **استفاده از قابلیت‌های (capabilities) به‌جای نام نقش‌ها**  
   به‌جای چک‌کردن نقش خاص، می‌توان بررسی کرد آیا کاربر توانایی خاص ادمین (`manage_options`، `edit_posts` سطح بالا و …) را دارد یا نه؛ این کار را در برابر تغییر نام/تعریف نقش‌ها مقاوم‌تر می‌کند.

2. **خروج سریع در ادمین یا درخواست‌های AJAX/REST**  
   در ابتدای تابع می‌توان نوشت:
   ```php
   if ( is_admin() || wp_doing_ajax() || (defined('REST_REQUEST') && REST_REQUEST) ) {
       return;
   }
   ```
   تا منطق فقط روی فرانت‌اند HTML اعمال شود.

3. **کَپسوله‌کردن منطق نقش**  
   اگر در پروژه چند جای دیگر هم لازم است این سه نقش را چک کنیم، بهتر است یک helper مثل `ez_is_front_user()` بسازیم و در آنجا شرط نقش را متمرکز کنیم؛ این تابع فقط آن helper را صدا بزند.
