# تابع `write_log`

## آدرس دقیق

- **فایل:** `wp-content/themes/escapezoom-v2/ahmadreza/init.php`
- **خط:** حدود ۳ تا ۱۲

## امضا

```php
function write_log($data)
```

## کار دقیق

- فقط وقتی `WP_DEBUG === true` باشد عمل می‌کند.
- اگر `$data` آرایه یا شیء باشد، با `print_r($data, true)` به رشته تبدیل و با `error_log()` چاپ می‌شود؛ وگرنه خود `$data` با `error_log()` چاپ می‌شود.
- برای دیباگ در محیط توسعه استفاده می‌شود.

## محل استفاده در سایت

- **تعریف:** فقط در `ahmadreza/init.php`.
- **فراخوانی:** در کد فعلی تم هیچ فراخوانی مستقیم `write_log(` در فایل‌های PHP یافت نشد؛ تابع در دسترس است ولی استفاده نشده.

## برای تغییر چه کار کنیم

1. **شرط لاگ:** برای فعال بودن در پروداکشن، شرط را به ثابت سفارشی (مثلاً `EZ_DEBUG_LOG`) یا آپشن تنظیمات تغییر دهید.
2. **محل خروجی:** به‌جای `error_log` می‌توان فایل لاگ اختصاصی یا پلاگین لاگر استفاده کرد.
3. **نام تابع:** نام عمومی است؛ با پیشوند تم (مثلاً `ez_write_log`) تداخل با پلاگین‌ها کمتر می‌شود.

## تابع/کد مشابه در سایت

- در تم از `error_log(...)` مستقیم در چند جا استفاده شده (مثلاً `inc/saeed-codes.php`, `inc/api-shortener.php`). تابع مشابه با نام دیگر در saeed-codes تعریف نشده.
- **ادغام:** می‌توان یک تابع مرکزی مثل `ez_write_log($data, $context = '')` در هلپر مشترک (مثلاً `app/functions/helper/debug.php`) تعریف و در همه جا استفاده کرد.

## بهینه‌سازی

1. پیشوند نام (مثلاً `ez_write_log`).
2. پارامتر اختیاری سطح لاگ (debug/info/error).
3. جایگزینی فراخوانی‌های پراکنده `error_log` با همین تابع برای مدیریت متمرکز.
4. برای شیءها استفاده از `wp_json_encode` با گزینه خوانا برای خروجی یکدست.
